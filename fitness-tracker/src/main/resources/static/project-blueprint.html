4b 現有 vs 建議 比較
❌ 現有設計（3張表）
  • ⚠️ 無 user 表 → 只能單人用
  • ⚠️ exercise_name 自由文字 → 拼字不一致難統計
  • ⚠️ body_part 放在 session → 一次訓多部位資訊丟失
  • ⚠️ body_weight 孤立 → 無法知道是誰的體重
✅ 建議設計（5張表）
  • ✅ user 表 → 支援多用戶
  • ✅ exercise 表 → 動作名稱標準化
  • ✅ body_part 移到 exercise → 精確到動作層級
  • ✅ user_id FK → 所有資料都歸屬用戶
升級策略：Phase 1 繼續用 3 張表做 MVP， Phase 2 再 migration 加入 user 和 exercise 表， 不需要重寫核心邏輯，只需加 FK。

8 設計決策討論（為什麼這樣建議）

① 為什麼加 user 表？

現在只有你一個人用，不需要登入。
但 一旦想讓朋友用、或部署到網路上，沒有 user 表就完全無法區分資料。
現在設計好 FK，Phase 2 加登入時不用改動核心邏輯。

② 為什麼加 exercise 表？

現在 exercise_name 是自由文字。
"臥推" vs "bench press" vs "Bench Press" 是 三筆不同資料，無法統計進步。
有 exercise 表才能做 "臥推 1RM 進步曲線"。

③ 為什麼 body_part 從 session 移到 exercise？

一次訓練常同時練多個部位（胸 + 肩 + 三頭）。
放在 session 代表整個 session 只有一個部位，資訊不夠精確。
放在 exercise（動作本身）才是正確的語義。

④ 為什麼現在不急著改？

MVP 原則：先讓功能跑起來，再優化結構。
Phase 1 繼續用現有 3 張表，Phase 2 做 DB migration 升級。
提早抽象 = 過度設計，浪費時間在看不見的東西上。

✅ 已處理

Thymeleaf 自動 HTML Escape 防 XSS

⚠️ Phase 2 需處理

Spring Security 加入路由保護
密碼必須 BCrypt 加密儲存
用戶只能看自己的資料（授權檢查）

⚠️ Phase 3 需處理

JWT 或 Session Token 時效管理
API Rate Limiting 防濫用
HTTPS 設定（部署階段）

CSRF（Thymeleaf 表單已自動處理）
OAuth2（個人 App 不需要）
2FA（超出 MVP 範圍）
