2
exercise 表 · 欄位說明

🏷️ name（動作名稱）

設為 UNIQUE，確保同一動作不會重複寫入。
前端文字搜尋 LIKE %keyword% 用這個欄位查。

🎯 body_part（訓練部位）

6 個部位：胸、背、腿、肩、手臂、核心
前端可以用部位做 Tab 篩選，縮小下拉選單範圍。

💡 category（動作分類）

COMPOUND（複合）：多關節，例如深蹲、臥推
ISOLATION（孤立）：單關節，例如二頭彎舉
未來可用來篩選適合當天的訓練類型。

🔒 preset（是否為預設）

true：系統預設的 50 個動作（啟動時自動寫入）
false：用戶自己新增的客製化動作
未來可讓預設動作不可刪除、客製動作可刪除。

exercise API 設計（3 支）
MethodEndpoint說明範例
GET /api/exercises 查全部動作（下拉選單用） → 53 筆
GET /api/exercises?bodyPart=胸 依部位篩選 → 9 筆
GET /api/exercises?q=臥推 文字模糊搜尋 → 臥推 / 啞鈴臥推...
POST /api/exercises 新增用戶自訂動作 preset=false
POST body 範例：
{ "name": "懸吊飛鳥", "bodyPart": "胸", "category": "ISOLATION" }
409 Conflict = 動作名稱已存在（不會重複新增）

6
設計決策說明 · 為什麼這樣做

① 為什麼 exercise 是獨立表，而不是 Enum 或 hardcode list？

用資料表的好處：之後可以讓用戶自己新增客製化動作（preset=false），不需要改程式碼。
Enum 的話每次加新動作都要重新部署，資料表只要 INSERT 一筆就好。
而且資料在 DB 裡，前端 API 查詢時可以做分頁、篩選、模糊搜尋，Enum 做不到。

② 為什麼 exercise 表現在還沒有 FK 連到 workout_set？

Phase 1 的目標是「快速讓功能跑起來」。如果現在就加 FK，要同時改 WorkoutSet entity、 WorkoutService 的存取邏輯、所有 DTO、前端表單……工程量很大。
現在的做法：exercise 表提供正確的動作名稱給前端選，選到的名稱寫進 exercise_name 字串欄位。 Phase 2 再做 DB migration 把字串升級為 FK，核心邏輯不用大改。

③ 為什麼 preset=true 的動作不讓用戶刪除？

系統預設動作是所有用戶共用的「字典」。如果讓用戶刪掉「深蹲」， 其他已有「深蹲」紀錄的 workout_set 資料就會對不上（未來加 FK 後更嚴重）。
preset=false 的自訂動作只有自己的資料在用，刪除比較安全。
